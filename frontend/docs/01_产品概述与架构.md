# 01 产品概述与架构

**版本号：** V2.2  
**更新日期：** 2026-03-21

---

## 一、产品概述



### 1.1 业务背景



MDM 控制中台需要支撑多租户运营场景，支持多个独立企业/组织共用同一套系统实例，同时保证数据完全隔离。多租户架构是 SaaS 化运营的基础，支撑未来按套餐分级计费、品牌白标、渠道代理等商业模式。



### 1.2 设计目标



| 目标 | 描述 |

|------|------|

| **租户强隔离** | 租户间数据完全隔离，跨租户访问在数据库层即被拦截 |

| **运营友好** | 超级管理员可一站式管理所有租户，含审核、套餐、配置 |

| **租户自治** | 各租户管理员可自主管理本单位用户、部门、门店 |

| **权限分级** | 支持超级管理员 → 租户管理员 → 普通用户三级权限体系 |

| **可扩展套餐** | 套餐与功能配额解耦，支持未来新增套餐类型 |



### 1.3 专业术语表



| 术语 | 说明 |

|------|------|

| 租户（Tenant） | 使用系统的企业/组织单位，拥有独立的数据空间和管理员 |

| 租户管理员 | 租户内最高权限用户，可管理本单位用户、部门、门店 |

| 超级管理员 | 平台运营方，可管理所有租户和平台级配置 |

| 套餐（Plan） | 租户订阅的功能等级，决定可用功能配额 |

| 单位（Unit） | 租户内的组织单元，可为部门或门店 |

| 数据隔离层级 | 租户维度（强制）→ 部门维度（可选）→ 角色维度（功能权限） |

## 二、整体架构



### 2.1 多租户隔离策略



采用 **Shared Database + tenant_id 字段隔离**（共享数据库 + 租户字段隔离）策略：



```

┌─────────────────────────────────────────────────────┐

│                   PostgreSQL                          │

│                                                      │

│  ┌──────────────────────────────────────────────┐    │

│  │              public schema                    │    │

│  │                                              │    │

│  │  平台级表: tenants, plans, dicts, audit_logs │    │

│  │                                              │    │

│  │  业务表(含 tenant_id):                       │    │

│  │    - users (tenant_id)                       │    │

│  │    - devices (tenant_id)                      │    │

│  │    - departments (tenant_id)                  │    │

│  │    - stores (tenant_id)                       │    │

│  │    - members (tenant_id)                      │    │

│  │    - tenant_quotas                             │    │

│  │                                              │    │

│  └──────────────────────────────────────────────┘    │

└─────────────────────────────────────────────────────┘

```



- **平台级表**（public schema）：租户台账、套餐定义、公共字典、审计日志

- **业务表**：所有业务表统一存放在 public schema，通过 `tenant_id` 字段实现租户隔离

- **强制隔离**：所有业务查询通过 GORM 中间件自动注入 `WHERE tenant_id = ?` 条件

- **跨租户查询**：仅超级管理员可访问，绕过租户隔离（内部接口，加严格审计）



#### GORM 中间件自动注入机制



```go

// GORM Callback 实现租户上下文自动注入

func TenantScopeInterceptor(db *gorm.DB, tenantID string) *gorm.DB {

    return db.Where("tenant_id = ?", tenantID)

}



// 所有业务表查询自动附带 tenant_id 条件

// 业务代码: db.Where("status = ?", "active").Find(&users)

// 自动转换为: db.Where("tenant_id = ? AND status = ?", tenantID, "active").Find(&users)

```

### UI规范

> **功能路径**：全局规范
> **权限要求**：所有用户
> **套餐要求**：所有套餐
> **规范版本**：V2.2（引用 `07_UI设计规范.md`）

#### 按钮命名规范

所有按钮必须使用「」双尖括号包裹中文名称，禁止使用英文或无包裹格式。

**正确示例：**
- 「搜索」「刷新」「新增用户」「编辑」「删除」「保存」「取消」「导出」「导入」

**错误示例：**
- ❌ `[搜索]`、`Search`、`search-btn`、`新增用户`

#### 按钮类型定义

| 按钮类型 | 视觉样式 | 使用场景 |
|----------|----------|----------|
| 主按钮 | 实心蓝色/品牌色 | 主要操作，如「保存」「提交」「通过」 |
| 次按钮 | 边框样式 | 次要操作，如「取消」「重置」「返回」 |
| 危险按钮 | 红色实心/边框 | 危险操作，如「删除」「禁用」「拒绝」 |
| 文字按钮 | 无边框 | 低强调操作，如「查看详情」「编辑」 |

#### 按钮权限要求

| 权限标识 | 说明 | 可见按钮 |
|----------|------|----------|
| all | 所有登录用户 | 搜索、刷新、查看详情 |
| tenant_admin | 租户管理员 | 含新建、编辑、删除、配置等全部写操作 |
| super_admin | 超级管理员 | 平台级管理，如删除租户、禁用用户 |

#### 按钮状态

| 状态 | 说明 | 视觉表现 |
|------|------|----------|
| normal | 默认状态 | 正常颜色，可点击 |
| hover | 悬停状态 | 颜色加深，显示手型光标 |
| loading | 加载状态 | 按钮禁用，显示加载图标 |
| disabled | 禁用状态 | 颜色变灰，不可点击 |

#### 页面布局（ArcoDesign Pro V2.2 列表页布局）

每个列表页面必须遵循以下统一的操作栏布局规范：

**布局结构：**
```

┌──────────────────────────────────────────────────────────────────────┐
│  工作台 / 组织管理 / 公司管理              (面包屑，左上角)           │
│                                                                      │
│  🔍 搜索...                                   (搜索框，左对齐，280px)  │
│                                                                      │
│  「新建」  「编辑」  「删除」   │   「批量导入」  「导出」  🔄        │
│  (主操作按钮组，左对齐)          (辅助操作按钮组)                     │
│────────────────────────────────────────────────────────────────────│
│  □全选  编号  │ 名称(可点击) │ 类型    │ 状态  │ 时间    │ 操作     │
│────────────────────────────────────────────────────────────────────│
│  ☐        001  │ [项目名称]  │ 运行中 │ ●在线 │ 2小时   │          │
│  ☐        002  │ [项目名称]  │ 已停止 │ ○离线 │ 3天     │          │
└────────────────────────────────────────────────────────────────────┘

```

**布局要点：**
1. **面包屑**：页面最左上角，格式：首页 / 模块名 / 页面名
2. **搜索框**：面包屑下方，左对齐，宽度约 280px，支持关键字筛选
3. **按钮分组**：
   - **主操作按钮组**（左）：「新建」「编辑」「删除」紧靠在一起
   - **辅助操作按钮组**：「批量导入」「导出」「刷新」挨在一起
   - 两组之间用 `│` 分隔
4. **数据表格**：
   - 每行左侧有复选框（支持全选），宽度 40px
   - **名称/id 可点击**：点击名称或编号打开详情弹窗，不在表格行内显示编辑/删除按钮
   - 操作列不再固定显示编辑/删除
5. **详情弹窗**：footer 只显示「取消」按钮

**规则：**
- 面包屑在最顶部，显示当前页面路径
- 搜索框位于面包屑下方，左对齐，宽度约 280px
- 主操作按钮：「新建」「编辑」「删除」靠左排列，间距 8px
- 辅助操作按钮：「批量导入」「导出」「🔄刷新」靠在一起
- 按钮之间间距：8px
- 按钮高度：32px（标准）或 28px（紧凑）

#### 表格布局规范

| 元素 | 规范 |
|------|------|
| 列宽 | 最小 80px，最大自适应，超长文字省略 |
| 行高 | 标准 48px，紧凑 36px |
| 斑马纹 | 可选，建议复杂表格启用 |
| 复选框 | 多选表格第一列，宽度 40px |
| 操作列 | 固定右侧，最小宽度 120px |
| 空状态 | 居中显示图标+文字，无"暂无数据"纯文字 |

#### 弹窗规范（V2.2 新增详情弹窗规范）

| 弹窗类型 | 触发场景 | 尺寸 |
|----------|----------|------|
| 全屏模态（Modal） | 新建/编辑复杂表单（如新建租户、编辑用户） | 宽度 800~1200px，高度 80vh |
| 抽屉（Drawer） | 快速操作（如分配角色、筛选条件） | 宽度 400~600px |
| 确认对话框（Dialog） | 删除确认、状态切换确认 | 宽度 400px，固定居中 |
| 消息提示（Toast） | 操作结果反馈 | 居中浮现 |
| **详情弹窗** | 点击名称/id打开详情 | 宽度 600~800px，仅显示信息 |

**新建/编辑模态（全屏模态，风格D）：**
- 右上角：「×」关闭按钮 + 「保存」+「取消」
- 内容区：表单主体，滚动
- 底部：固定在模态框底部，「取消」在左，「保存」在右

**详情弹窗（V2.2 新增）：**
- 右上角：「×」关闭按钮
- 内容区：只读展示信息，滚动
- 底部：**仅显示「取消」按钮**，无保存/编辑按钮

**抽屉规范：**
- 从右侧滑入
- 右上角：「×」关闭按钮
- 宽度：400px（筛选）、500px（详情）

#### 交互规范

- **删除确认**：所有删除操作必须弹出确认对话框，显示待删除对象信息
- **Toast 反馈**：操作后顶部居中显示成功/失败提示，3~5秒自动消失
- **表单校验**：失焦时校验必填项和格式，错误信息显示在字段下方；提交时滚动到第一个错误处
- **加载状态**：首次加载显示骨架屏，提交时按钮显示 loading 禁止双击

#### 响应式规范（V2.2）

| 断点 | 屏幕宽度 | 布局变化 |
|------|----------|----------|
| 超大屏 | ≥1920px | 表格列全展示 |
| 大屏 | 1440~1919px | 标准布局 |
| 中屏 | 1024~1439px | 操作栏换行，按钮组分两行 |
| 小屏 | 768~1023px | 表格水平滚动 |
| 平板 | <768px | 强制收起操作栏 |
