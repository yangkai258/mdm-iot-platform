## 三、租户入驻审核



### 3.1 租户注册申请



#### 3.1.1 申请入口



- 独立注册页面：`/register`

- 申请表单，无需登录即可提交



#### 3.1.2 申请表单字段



| 字段名 | 类型 | 必填 | 校验规则 | 说明 |

|--------|------|------|----------|------|

| company_name | VARCHAR(100) | ✅ | 2-100字符，不得重复 | 企业/组织全称 |

| contact_name | VARCHAR(50) | ✅ | 2-50字符 | 联系人姓名 |

| contact_phone | VARCHAR(20) | ✅ | 手机号正则 `/^1[3-9]\d{9}$/` | 联系人手机号 |

| contact_email | VARCHAR(100) | ✅ | 邮箱格式校验 | 联系人邮箱 |

| industry | VARCHAR(50) | ❌ | 下拉选择 | 所属行业 |

| company_size | VARCHAR(20) | ❌ | 下拉选择 | 企业规模 |

| plan_id | INT | ✅ | 必须为有效套餐 ID | 选择的套餐 |

| use_case | TEXT | ❌ | 最多 500 字 | 使用场景简述 |



#### 3.1.3 业务规则



1. `company_name` 全局唯一，重复提交返回错误提示

2. `plan_id` 仅决定初始订阅意向，实际套餐以审核通过后配置为准

3. 提交后发送邮件/短信通知申请人："您的申请已提交，请等待审核"

4. 申请记录写入 `tenant_applications` 表，状态初始化为 `pending`

### 3.3 审核通过自动开通



> **重要说明**：开通流程**不创建独立 schema**，所有业务表通过 `tenant_id` 字段隔离。开通时只需在 `tenants` 表创建记录，初始化 `tenant_quotas` 配额记录。



审核通过后，系统自动执行以下开通流程：



| 步骤 | 操作 | 说明 |

|------|------|------|

| 1 | 在 `tenants` 表创建租户记录 | 插入租户基本信息，状态置为 `activating` |

| 2 | 在 `tenant_quotas` 表初始化配额记录 | 根据套餐配置初始化 user_count=0, device_count=0 等 |

| 3 | 创建租户管理员账号 | 根据申请信息创建第一个用户，角色为 `tenant_admin` |

| 4 | 发送欢迎邮件 | 包含管理员账号、初始密码（随机生成，需首次登录修改）、登录链接 |

| 5 | 更新申请状态 | 将 `tenant_applications` 状态更新为 `approved` |

| 6 | 更新租户状态为 `active` | 开通完成 |



> **异常处理**：若步骤 1-2 失败（如数据库连接超时），最多重试 3 次，间隔 10 秒。3 次后仍失败则状态标记为 `activation_failed`，发送告警给超级管理员。

### 5.2 用户角色体系



| 角色 | 英文标识 | 权限描述 |

|------|----------|----------|

| 租户管理员 | tenant_admin | 管理本单位用户、部门、门店、设备配置；不可见平台级功能 |

| 普通用户 | normal_user | 仅可使用被授权的设备功能，数据范围受部门/门店权限限制 |



**约束**：

- 每个租户至少有 1 个 `tenant_admin`

- 删除租户管理员前必须先转移权限

- 角色变更记录写入审计日志



### 5.3 配额校验机制



所有创建操作需先校验配额，超额返回错误。



#### 5.3.1 配额校验点



| 操作 | 校验配额 | 超额处理 |

|------|----------|----------|

| 创建用户 | `user_count < user_quota` | 返回错误码 `QUOTA_USER_EXCEEDED` |

| 注册设备 | `device_count < device_quota` | 返回错误码 `QUOTA_DEVICE_EXCEEDED` |

| 创建部门 | `dept_count < dept_quota` | 返回错误码 `QUOTA_DEPT_EXCEEDED` |

| 创建门店 | `store_count < store_quota` | 返回错误码 `QUOTA_STORE_EXCEEDED` |



#### 5.3.2 配额表设计



```sql

CREATE TABLE tenant_quotas (

    id SERIAL PRIMARY KEY,

    tenant_id UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,

    user_count INT DEFAULT 0,

    device_count INT DEFAULT 0,

    dept_count INT DEFAULT 0,

    store_count INT DEFAULT 0,

    updated_at TIMESTAMPTZ DEFAULT NOW(),

    UNIQUE(tenant_id)

);

```



#### 5.3.3 配额校验流程



```

1. 接收创建请求

2. 查询 tenant_quotas 获取当前配额使用情况

3. 对比套餐 plan.user_quota

4. 如已满额，返回错误:

   {

     "code": "QUOTA_USER_EXCEEDED",

     "message": "用户数已达上限（5/5），请联系管理员升级套餐",

     "upgrade_url": "/admin/tenants/{tenant_id}/change-plan"

   }

5. 如未满额，执行创建，原子更新 tenant_quotas 对应计数

```



### 5.4 跨租户隔离



**强制隔离原则**：

1. 所有业务表必须有 `tenant_id` 字段（UUID）

2. 中间件层自动注入 `tenant_id`，业务代码无需手动拼接

3. 超级管理员跨租户操作需额外 JWT claims 携带 `is_super_admin=true`，并记录审计日志

4. 租户管理员调用任意接口，返回数据严格限定在 `tenant_id` 范围内



**隔离示意**：



```

请求: GET /api/v1/devices

中间件处理:

  1. 从 JWT 提取 tenant_id = "T-002"

  2. GORM Scope 自动追加 WHERE tenant_id = 'T-002'

  3. 业务代码: db.Where("status = ?", "active").Find(&devices)

  4. 最终执行: SELECT * FROM devices WHERE tenant_id = 'T-002' AND status = 'active' ...

```

## 六、租户单位管理



### 6.1 组织架构树



租户内单位分为两级：**部门（Department）** 和 **门店（Store）**。



- 部门：虚拟组织单元，用于数据权限隔离（如按业务线划分）

- 门店：实体经营单位，关联会员管理系统（如零售门店、健身房等）



> 单位树最大深度：2 级（部门 → 门店），禁止跨级（部门下不能再挂部门）



#### 组织树结构



```

Tenant Root

└── 部门 A

    ├── 门店 A1

    ├── 门店 A2

└── 部门 B

    ├── 门店 B1

    └── 门店 B2

```

## 七、多维数据权限



### 7.1 三维权限体系



```

┌─────────────────────────────────────────────────────┐

│           权限维度 1: 租户维度 (Tenant)               │

│   所有业务数据以 tenant_id 为最高隔离级别             │

│   任何请求自动携带租户上下文，跨租户访问被直接拒绝    │

└─────────────────────────────────────────────────────┘

                         │

                         ▼

┌─────────────────────────────────────────────────────┐

│           权限维度 2: 部门维度 (Department)           │

│   数据权限：用户只能看到其所属部门及下级门店的数据     │

│   适用场景：按业务线/区域隔离数据                    │

│   超级管理员可见所有部门数据                          │

└─────────────────────────────────────────────────────┘

                         │

                         ▼

┌─────────────────────────────────────────────────────┐

│           权限维度 3: 角色维度 (Role)                 │

│   功能权限：角色决定可操作的菜单和 API               │

│   tenant_admin: 管理功能全开                          │

│   normal_user: 按需分配功能点                         │

└─────────────────────────────────────────────────────┘

```



### 7.2 租户维度（强制隔离）



| 规则 | 说明 |

|------|------|

| 隔离粒度 | 所有业务表 `tenant_id` 字段，GORM 中间件自动注入 |

| 中间件注入 | 请求进入时从 JWT 提取 `tenant_id`，注入到 Context，GORM Callback 自动拼接 WHERE 条件 |

| 数据库策略 | 租户管理员只能访问 `tenant_id` 对应的数据行 |

| 超级管理员 | 可指定 `tenant_id=*` 跨越隔离，需额外审计 |

| 禁止项 | 禁止不带 `tenant_id` 条件的 DELETE/UPDATE 全表操作 |



### 7.3 部门维度（数据权限）



| 规则 | 说明 |

|------|------|

| 可见范围 | 用户可见自己所属部门 + 所有下级门店的数据 |

| 数据过滤 | 所有查询自动追加 `dept_id IN (...)` 条件 |

| 跨部门查看 | 需租户管理员授权（配置用户可访问的额外部门） |

| 超级管理员 | `dept_id=*` 可见所有部门数据 |



#### 数据权限字段注入示例



```go

// 中间件: 基于用户 dept_ids 构建数据权限范围

ctx.Set("data_scope", buildDeptScope(user.DeptIDs))



// 查询时自动追加

query := query.Where("dept_id IN (?)", ctx.Get("data_scope"))

```



### 7.4 角色维度（功能权限）



| 权限点 | tenant_admin | normal_user |

|--------|--------------|-------------|

| 用户管理 | ✅ | ❌ |

| 部门管理 | ✅ | ❌ |

| 门店管理 | ✅ | ❌（仅查看） |

| 设备管理 | ✅ | 按部门授权 |

| OTA 升级 | ✅ | ❌ |

| 固件管理 | ✅ | ❌ |

| 数据导出 | ✅ | 按套餐 |

| 租户配置 | ✅ | ❌ |



#### 权限控制实现



```

1. JWT claims 包含: { user_id, tenant_id, role, dept_ids[], permissions[] }

2. 路由中间件: 校验 permissions 列表中是否包含目标 API

3. 按钮级别: 前端根据 permissions 渲染显示/隐藏

4. 后端校验: 所有写操作二次校验权限点

```
