# MDM 控制中台 - 产品深度分析报告

**版本：** V1.1  
**分析日期：** 2026-03-20  
**分析角色：** 产品经理 (agentcp)  
**项目阶段：** 基础功能已实现，但比较粗糙

## 一、当前功能矩阵

### 1.1 设备管理模块

| 功能点 | API 路径 | 后端状态 | 前端状态 | 说明 |
|--------|----------|:--------:|:--------:|------|
| 设备注册 | POST /api/v1/devices/register | ✅ 已实现 | N/A | 仅校验MAC格式，未校验重复device_id |
| 设备列表 | GET /api/v1/devices | ✅ 已实现 | ✅ 已实现 | 分页+筛选，但在线状态筛选为内存过滤效率低 |
| 设备详情 | GET /api/v1/devices/:device_id | ✅ 已实现 | ✅ 已实现 | 包含shadow和pet_profile |
| 设备绑定 | POST /api/v1/devices/bind/:sn_code | ✅ 已实现 | ⚠️ 缺失 | 路径用sn_code但设备唯一标识是device_id |
| 设备解绑 | POST /api/v1/devices/unbind/:sn_code | ✅ 已实现 | ⚠️ 缺失 | 同上 |
| 状态更新 | PUT /api/v1/devices/:device_id/status | ✅ 已实现 | ⚠️ 缺失 | 5状态流转 |
| 设备删除 | DELETE /api/v1/devices/:device_id | ✅ 已实现 | ⚠️ 缺失 | 软删除 |
| 宠物配置 | GET/PUT /api/v1/devices/:device_id/profile | ✅ 已实现 | ✅ 已实现 | 基础宠物配置 |

**问题发现：**
- 设备绑定/解绑用 `sn_code` 作为路径参数，与其他设备接口用 `device_id` 不一致
- 设备列表的 `status`（在线/离线）筛选是内存过滤，大数据量下性能差
- 设备注册时 `device_id` 为 optional，但 `mac_address` 已唯一索引，存在业务逻辑漏洞（同一MAC可多次注册？）

## 二、问题清单

### 🔴 P0 - 阻断性问题（系统不可用/核心流程断裂）

1. **OTA升级是死代码** - 发布任务创建后无后台Worker执行，设备永远收不到升级指令
2. **指令下发MQTT为nil** - 所有设备控制指令实际无法下发，设备完全失控
3. **告警检查从未触发** - `CheckAlerts()` 定义了但从未调用，告警系统形同虚设
4. **Dashboard统计SQL错误** - `stats.OnlineDevices` 查询不存在的字段，永远为0
5. **会员等级自动升级无执行器** - 规则表存在但定时任务缺失，无法自动升级

### 🟡 P1 - 严重问题（核心功能不完整）

6. **设备绑定/解绑路径不一致** - 部分接口用sn_code，部分用device_id，容易混淆
7. **积分规则引擎缺失** - 积分获取/抵扣无计算逻辑，PointsRule表是摆设
8. **优惠券发放/核销流程缺失** - CouponGrant表存在但无业务逻辑
9. **促销规则引擎缺失** - Promotion.rule_config是JSONB但无解析执行代码
10. **无实时状态推送** - 设备状态变化需要前端刷新或轮询，用户体验差
11. **会员编号不统一** - 代码未强制 `HY+日期+6位序号` 格式
12. **JWT Token无登录接口** - 中间件存在但无产生token的/login接口
13. **设备注册逻辑漏洞** - device_id为optional且MAC重复注册时只更新固件版本，不校验业务状态

### 🟠 P2 - 体验问题（功能粗糙但可工作）

14. **OTA固件包无文件上传** - 仅存URL，无本地上传能力
15. **OTA发布任务无管理界面** - 前端无任何OTA任务管理页面
16. **指令下发无前端界面** - 无命令发送的UI
17. **组织架构无前端页面** - 后端CRUD全了但前端几乎没有
18. **会员管理前端粗糙** - 无表单验证、无操作反馈、无批量操作
19. **告警规则类型单一** - 仅支持battery_low和offline
20. **告警无通知渠道** - 定义了notify_ways字段但无任何实现
21. **设备列表无导出** - 无Excel/CSV导出功能
22. **无批量设备操作** - 批量状态更新、批量解绑等
23. **API响应格式不统一** - 部分返回`code/message/data`，部分直接返回对象

## 三、JTBD 分析

### 3.1 用户角色识别

| 角色 | 核心任务 | 痛点 |
|------|----------|------|
| **设备运维人员** | 监控设备状态、快速定位故障设备、执行OTA升级 | 无法实时感知设备异常，OTA升级无法确认是否成功 |
| **设备管理员** | 管理设备台账、绑定解绑设备、配置宠物参数 | 设备列表信息不全，绑定流程繁琐 |
| **运营主管** | 管理会员、分析会员数据、发放优惠券 | 会员数据分散，优惠券发放靠手工 |
| **客服人员** | 查询会员信息、调整积分、处理投诉 | 无快速查询入口，积分调整无记录 |

### 3.2 JTBD 核心任务分析

#### 任务1：确保设备正常运行
**用户真正想要：** 我的设备都在线吗？有任何设备出问题了吗？
**当前实现：** 设备列表有在线状态，但无实时推送，需要人工刷新
**缺失：** 实时Dashboard + 告警推送

#### 任务2：安全可控地升级设备固件
**用户真正想要：** 选一批设备，升级到新版本，看到升级进度，失败自动回滚
**当前实现：** 固件包录入 → 创建任务 → 没了（实际不执行）
**缺失：** 完整OTA执行流水线 + 进度可视化 + 自动暂停/回滚

#### 任务3：远程控制设备
**用户真正想要：** 看到设备当前状态，发送指令，确认指令被执行
**当前实现：** 有指令下发API，但MQTT为nil，实际无法执行
**缺失：** MQTT集成 + 指令状态追踪

#### 任务4：精准营销会员
**用户真正想要：** 筛选一批会员，发送优惠券，看到核销率
**当前实现：** 优惠券CRUD全了，但没有发放和核销流程
**缺失：** 发放规则引擎 + 核销接口 + 效果统计

#### 任务5：自动化会员运营
**用户真正想要：** 消费后自动积分，积分够自动升级，等级变化发通知
**当前实现：** 所有规则表都建了，但执行器全无
**缺失：** 规则引擎 + 定时任务 + 事件触发机制
