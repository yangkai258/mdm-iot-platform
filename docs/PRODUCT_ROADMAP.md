## 一、项目概述

### 1.1 产品定位

MDM (Mobile Device Management) 控制中台是"端云协同"AI 电子宠物系统的核心管理平台，负责统一管理分布在全球的 M5Stack 硬件终端，维护设备全生命周期（注册、绑定、监控、OTA 升级、指令控制），并通过会员管理模块支持用户运营能力。

### 1.2 技术栈

| 层级 | 技术选型 |
|------|----------|
| 设备端 | M5Stack 硬件 + MimiClaw 固件 |
| 通信层 | MQTT (paho.mqtt.golang) + EMQX 5.0 |
| 后端 | Go + Gin + GORM + PostgreSQL + Redis |
| 前端 | Vue 3 + TypeScript + Vite + Arco Design Vue |
| 部署 | Docker + docker-compose + Nginx |

### 1.3 核心业务流程

```
M5Stack 设备
    │ MQTT /device/{id}/up/status (心跳)
    │ MQTT /device/{id}/up/ota_progress (OTA进度)
    ▼
EMQX Broker
    │
    ▼
MDM Backend (Go+Gin)
    ├── 设备注册/绑定/指令下发
    ├── OTA Worker (后台轮询部署任务)
    ├── 告警检测 (CheckAlerts)
    └── 会员管理
            │
            ▼
    Vue 3 管理控制台 / 设备端 App
```

#### 模块 A：OTA 固件升级

**优先级：P0（最高）**

**现状分析：**

| 子功能 | 代码状态 | 说明 |
|--------|----------|------|
| 固件包 CRUD | ⚠️ 部分完成 | `ota_package_controller.go` 实现了完整 CRUD，但 models 层存在字段不一致（PRD 用 `version`，代码用 `VersionCode`） |
| 部署任务 CRUD | ⚠️ 部分完成 | `ota_controller.go` 有 `CreateDeployment`，但字段与 PRD 不匹配 |
| 灰度策略 | ⚠️ 部分完成 | 百分比/白名单/全量字段在 model 和 controller 中名称不一致 |
| OTA Worker | 🔜 未开始 | **最大缺口** — 后台 Worker 未实现，部署任务无法自动下发 |
| 设备升级进度追踪 | ⚠️ 部分完成 | `OTAProgress` model 和部分接口存在，设备端上报回调未实现 |
| 设备端 OTA 回调 | 🔜 未开始 | `/api/v1/ota/devices/:device_id/check` 和 `/report` 未在路由注册 |

**技术依赖图：**

```
OTA Worker ← 依赖 → OTA 数据库模型对齐
          ← 依赖 → MQTT handler (指令下发)
          ← 依赖 → 设备端固件支持 OTA 协议
```

**开发成本：**
- 模型对齐：1 天
- OTA Worker 实现：3 天
- 设备端回调接口：1 天
- 前端部署任务管理 UI：2 天
- 测试与联调：2 天
- **合计：约 9 个工作日**

**商业价值：**
- 无 OTA 能力 → 设备无法远程修复 bug / 迭代功能 → 产品不可用
- **P0 阻断性**

## 七、验收标准

### Phase 1 验收标准

| 模块 | 验收标准 |
|------|----------|
| OTA Worker | 1. 创建部署任务后，Worker 自动选中目标设备并下发 MQTT OTA 指令<br>2. 设备上报进度后，OTAProgress 表正确更新<br>3. 成功率 < 80% 时，部署任务自动暂停并记录原因 |
| OTA 模型对齐 | 1. 所有接口返回字段与 PRD 定义一致<br>2. 数据库字段与 models/go 对齐，无遗露字段 |
| 告警触发 | 1. 设备电量 < 15% 时，自动创建告警记录<br>2. 设备离线 > 90 秒时，自动创建告警记录<br>3. 告警记录包含 device_id / alert_type / trigger_val / threshold |
| 设备端回调 | 1. 设备 GET /ota/devices/:id/check 返回正确格式<br>2. 设备 POST /ota/devices/:id/report 正确更新进度 |

### Phase 2 验收标准

| 模块 | 验收标准 |
|------|----------|
| 告警通知 | 1. 告警触发后，根据 notify_ways 发送邮件/Webhook<br>2. 告警支持确认/解决流程 |
| 宠物配置 | 1. 运营人员在控制台设置免打扰时间，设备端收到并应用<br>2. 宠物性格/交互频率正确下发到设备 |
| 会员管理 | 满足 `MEMBER_REQUIREMENTS.md` 中 P0/P1 功能定义 |
