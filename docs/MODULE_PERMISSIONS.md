# 模块 PRD：权限管理 (Permissions)

**版本：** V1.0
**模块负责人：** agentcp
**编制日期：** 2026-03-21

---

## 1. 概述

权限管理模块为 MDM 中台提供基于角色的访问控制（RBAC），支持功能权限和数据权限的精细化管控，确保不同角色用户只能访问其授权范围内的资源和数据。

**业务目标：**
- 基于角色的功能权限控制（菜单/按钮/API）
- 数据权限控制（组织/设备/业务数据隔离）
- 权限分配可视化配置
- 角色权限矩阵清晰呈现

## 3. 数据模型

### 3.1 权限表 (permissions)

| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| id | uint | 主键 | PK |
| code | string | 权限代码 | unique, not null |
| name | string | 权限名称 | not null |
| type | string | permission/menu/button/api | not null |
| parent_id | uint | 父级ID | FK → permissions |
| path | string | 菜单路径/API路径 | - |
| sort_order | int | 排序 | default 0 |
| created_at | datetime | 创建时间 | auto |

**type 说明：**
- `permission`：功能权限（顶层模块）
- `menu`：菜单项
- `button`：按钮权限
- `api`：API接口权限

### 3.2 角色表 (roles)

| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| id | uint | 主键 | PK |
| name | string | 角色名称 | not null |
| code | string | 角色代码 | unique, not null |
| description | string | 描述 | - |
| status | int | 1=启用 0=禁用 | default 1 |
| is_system | bool | 是否系统内置角色 | default false |
| created_by | string | 创建人 | not null |
| created_at | datetime | 创建时间 | auto |
| updated_at | datetime | 更新时间 | auto |

**系统内置角色（is_system=true，不可删除）：**
- `super_admin`：超级管理员，拥有所有权限
- `device_admin`：设备管理员
- `ops_admin`：运营管理员
- `viewer`：只读用户

### 3.3 角色权限关联表 (role_permissions)

| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| id | uint | 主键 | PK |
| role_id | uint | 角色ID | FK → roles |
| permission_id | uint | 权限ID | FK → permissions |
| data_scope | string | 数据权限范围 | default 'all' |
| created_at | datetime | 创建时间 | auto |

**data_scope 取值：**
- `all`：全部数据
- `org`：所属组织数据
- `org_and_children`：所属组织及下级组织数据
- `self`：仅本人数据
- `custom`：自定义数据范围（配合 org_ids 字段）

### 3.4 用户角色关联表 (user_roles)

| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| id | uint | 主键 | PK |
| user_id | string | 用户ID | index |
| role_id | uint | 角色ID | FK → roles |
| created_at | datetime | 创建时间 | auto |

### 3.5 数据权限配置表 (data_scope_config)

| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| id | uint | 主键 | PK |
| role_id | uint | 角色ID | FK → roles |
| resource_type | string | 资源类型 device/organization/app | - |
| org_ids | string | 授权的组织ID列表，逗号分隔 | - |
| created_at | datetime | 创建时间 | auto |

## 4. 接口定义

### 4.1 角色管理

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/v1/roles | 角色列表 |
| POST | /api/v1/roles | 创建角色（name, code, description, status） |
| PUT | /api/v1/roles/:id | 更新角色 |
| PUT | /api/v1/roles/:id/status | 启用/禁用角色（body: {status: 1/0}） |
| DELETE | /api/v1/roles/:id | 删除角色（is_system=true 不可删除） |
| POST | /api/v1/roles/:id/copy | 复制角色（body: {new_name, new_code}） |

### 4.2 权限分配

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/v1/roles/:id/permissions | 获取角色权限列表 |
| PUT | /api/v1/roles/:id/permissions | 分配权限（body: {permissions: [{id, data_scope}], org_ids: []}） |
| GET | /api/v1/permissions/tree | 获取权限树（所有权限，含菜单结构） |

**分配权限请求体：**
```json
{
  "permissions": [
    { "id": 1, "data_scope": "all" },
    { "id": 2, "data_scope": "org" },
    { "id": 3, "data_scope": "self" }
  ],
  "org_ids": [1, 2, 3]
}
```

### 4.3 用户角色绑定

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/v1/users/:id/roles | 获取用户角色列表 |
| PUT | /api/v1/users/:id/roles | 绑定角色（body: {role_ids: [1, 2]}） |

### 4.4 权限验证

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/v1/auth/permissions | 获取当前用户权限（含数据范围） |
| GET | /api/v1/auth/menus | 获取当前用户可见菜单 |
| POST | /api/v1/auth/check | 验证权限（body: {permission_code}） |

**响应示例（GET /auth/permissions）：**
```json
{
  "user_id": "user_001",
  "roles": ["device_admin", "viewer"],
  "permissions": [
    { "code": "device:view", "name": "查看设备", "type": "button", "data_scope": "org" },
    { "code": "device:manage", "name": "管理设备", "type": "button", "data_scope": "org" },
    { "code": "ota:view", "name": "查看OTA", "type": "menu", "data_scope": "all" }
  ],
  "data_scope": {
    "device": "org_and_children",
    "org_ids": [1, 2, 3]
  }
}
```

## 5. 流程图

### 5.1 数据权限控制流程

```
请求进入 API Gateway
    │
    ▼
JWT Token 解析 → 获取 user_id
    │
    ▼
查询用户角色 → user_roles → role_ids
    │
    ▼
查询角色权限 → role_permissions → permissions + data_scope
    │
    ▼
合并所有权限（含 data_scope）：
    │
    ├─► 数据范围取并集
    │       规则：取所有角色中数据范围最大的
    │       例：role_A=org, role_B=org_and_children → 最终=org_and_children
    │
    └─► 自定义 org_ids：合并所有角色的 org_ids
    │
    ▼
数据查询时注入权限过滤条件：
    │
    ├─► data_scope='all' ── 无过滤，返回全部
    │
    ├─► data_scope='org' ── WHERE org_id = :user_org_id
    │
    ├─► data_scope='org_and_children' ── WHERE org_id IN (:user_org_id, :children_ids)
    │
    └─► data_scope='self' ── WHERE created_by = :user_id
    │
    ▼
返回过滤后的数据
```

### 5.2 权限验证中间件流程

```
请求: POST /api/v1/devices/:id
    │
    ▼
PermissionMiddleware
    │
    ▼
检查是否为系统内置路径（/auth/*, /public/*）── 是 ──► 放行
    │
    ▼
获取请求所需权限码：route config → "device:manage"
    │
    ▼
查询用户权限：user_permissions 缓存
    │
    ├─► 命中 ──► 有权限？── 是 ──► 放行
    │                              │
    │                         否 ──► 返回 403
    │
    └─► 未命中 ──► 从 DB 加载并缓存（Redis, TTL 5min）
    │
    ▼
返回 403 Forbidden
```

## 7. 验收标准

### P0 验收标准

| 用例 | 验收条件 | 测试方法 |
|------|----------|----------|
| 角色创建 | 创建角色后权限树可见 | 创建并验证 |
| 权限分配 | 分配后用户立即拥有对应权限 | 绑定角色后验证 API 访问 |
| 功能权限验证 | 无权限用户访问返回 403 | 用无权限账号访问 |
| 数据权限隔离 | org 范围用户看不到其他 org 数据 | 不同 org 用户交叉验证 |
| 超级管理员 | super_admin 拥有所有权限 | 验证 bypass 所有检查 |

### P1 验收标准

| 用例 | 验收条件 | 测试方法 |
|------|----------|----------|
| 角色禁用 | 禁用后该角色用户无法登录 | 禁用角色后测试 |
| 角色复制 | 复制后新角色拥有相同权限 | 复制并对比 |
| 权限变更审计 | 变更写入审计日志 | 查看审计记录 |
| 多角色叠加 | 用户多角色权限取并集 | 绑定多个角色验证 |

| OTA升级 | ota:deploy | ✅ | ✅ | ❌ | ❌ |
| 告警管理 | alert:view | ✅ | ✅ | ✅ | ✅ |
| 告警管理 | alert:handle | ✅ | ✅ | ✅ | ❌ |
| 告警管理 | alert:delete | ✅ | ❌ | ✅ | ❌ |
| 系统管理 | user:view | ✅ | ❌ | ❌ | ❌ |
| 系统管理 | user:manage | ✅ | ❌ | ❌ | ❌ |
| 系统管理 | role:view | ✅ | ❌ | ❌ | ❌ |
